# 长期用户视角：优化意见与方向

> 模拟一个连续使用 3-4 周的真实用户视角，对当前产品提出的优化建议。

---

## 一、数据真实性：MOCK_LLM=true 让产品形同虚设

**现状**：`.env` 中 `MOCK_LLM=true` 意味着每天看到的是同一段硬编码数据——"8个AI项目、6个DevTools、curious_tilting、技术圈不温不火"。

**用户心理路径**：
- 第一天：猫好可爱，数据也有意思
- 第三天：怎么感觉跟昨天差不多？点 `?debug=1` 切换状态玩一玩
- 第五天：发现数据根本没变过，猫在骗我，再也不打开了

**方向**：
- 生产环境默认 `MOCK_LLM=false`
- LLM 调用失败时不要直接返回固定 mock，而是回退到**昨日真实数据**，并在 UI 标注"数据更新失败，显示昨日快照"
- 增加 API 调用失败的重试机制（指数退避，最多 3 次）

---

## 二、核心体验疲劳：8 种离散状态撑不过两周

**现状**：猫的状态被映射到 8 个离散 bucket，动画通过 CSS class 切换。这是产品层面最大的留存问题。

**用户心理路径**：
- 第一周：新鲜，每种状态都想看看，甚至会故意刷 `?debug=1` 切换着玩
- 第二周：开始意识到"怎么又是 excited_bouncing，最近天天 AI 爆发"
- 第三周：猫的动画已经全看过了，没有新东西，打开频率从每天降到每周
- 第四周：不打开了

**核心矛盾**：真实世界是连续的，但猫只有 8 张脸。AI 占比 41% 和 AI 占比 70% 都是 `excited_bouncing`，动画毫无区别。

**方向**：
1. **给状态加"强度"**：同样是 `excited_bouncing`，AI 占 40% 时猫轻轻颠脚，占 70% 时猫弹到飞起。用 CSS 变量 `--intensity` 控制动画幅度（amplitude = 占比 / 阈值）
2. **增加随机微表情层**：在状态动画之上叠加独立的小动画——耳朵偶尔自己抖一下、尾巴无规律摆一下、眼睛随机眨。让同一状态每天看起来也不完全一样
3. **猫有"记忆"**：LLM 分析时注入昨天的 `cat_state` + `narrative`，猫可以说"昨天还说 AI 要降温，今天又炸了，两脚兽你们说话不算数喵"

---

## 三、最致命的缺失：没有历史趋势页面

**现状**：`/api/history` 接口只返回 `date + cat_state + narrative`，前端完全没有调用它。用户每天只能看到孤立的"今天"，看不到任何纵向对比。

**但对于长期用户，最有价值的恰恰是时间轴**：
- "这周 AI 项目连续 5 天占比超 40%——这波浪潮是真的"
- "上次 security 集中爆发还是上个月那波漏洞事件，又来了"
- "猫这周心情日记：周一困→周二炸毛→周三专注→周四好奇→周五又困了"

**方向**：
1. **新增 `/api/trends?days=30` 接口**：返回每日的 categories 分布 + cat_state，支持画趋势图
2. **前端新增"猫的心情日记"横向时间轴**：每天一个小猫头像 icon，不同状态不同表情，点击可回溯当天详情
3. **在今日页面增加"对比昨日"delta 信息**：如"AI 项目 +3 · 总星数 +12k · 猫比昨天兴奋"，让数字有上下文才有意义

---

## 四、调度器设计不合理：每 4 小时浪费 API 额度

**现状**：`scheduler.py` 设置 `hours=4`，每 4 小时执行一次。GitHub Trending 每天只更新一次（北京时间约凌晨刷新），因此每天 6 次调用中 5 次拿到完全一样的数据，白白烧 DeepSeek API 费用。

**方向**：
- 改为每天定时执行一次（如北京时间 8:00，Trending 已更新完毕），使用 cron 表达式 `0 8 * * *`
- 增加失败重试：8:00 爬取失败则 30 分钟后重试，最多 3 次，不等到第二天
- 如果全天都失败了，保留昨天的数据并在 UI 标注"数据可能不是最新的"

---

## 五、LLM Prompt 的上下文盲区

**现状**：`analyzer.py` 的 SYSTEM_PROMPT 写得很有猫味，但它没有任何历史上下文。猫每天都是"第一次看 GitHub"，不会吐槽"连续三天 AI 刷屏了"。

**同时存在规则冗余**：Prompt 中既写了状态触发规则（AI>40%→bouncing），又让 LLM 自由判断。结果是 LLM 有时按规则走，有时自由发挥，分类稳定性差。

**方向**：
1. **分离规则与叙事**：分类和状态判定逻辑从 Prompt 中拿出来放到 Python 代码里——比例计算是确定性的，不需要 LLM 来判断。LLM 只负责两个任务：(a) 项目分类 (b) 写叙事。cat_state 由代码根据分类结果精确计算，保证一致性
2. **注入昨日上下文**：在 user prompt 末尾追加"昨天的状态：{cat_state}，昨天的叙述：{narrative}"，让猫的吐槽有连续性
3. **叙事多样化**：给 LLM 加一个随机 seed 或 style hint（"今天用吐槽风" / "今天用撒娇风"），避免每天叙事风格雷同

## 六、前端交互和移动端体验

**现状问题**：

| 问题 | 影响 |
|---|---|
| 加载态只有一个脉冲文字"正在窥探 GitHub..." | 冷启动 + LLM 调用需 5-8 秒，用户以为页面挂了 |
| 25 个仓库卡片一列排开，无法筛选 | 想找 Python 项目要肉眼扫 25 张卡片 |
| 仓库卡片点击直接跳转 GitHub `target=_blank` | 打断浏览体验，点完回不来了 |
| 错误提示只有一种"网络出了点问题" | 后端挂了 vs LLM 超时 vs GitHub 被墙，都是同一句话 |
| 移动端 280px 猫 + 卡片挤压 | 在小屏上猫占了半屏，卡片文字被压缩 |

**方向**：
1. **首次加载用猫的骨架屏**（SVG 猫轮廓 + shimmer 动画），比纯文字生动得多
2. **仓库卡片增加语言过滤栏**：提取所有语言去重后做成可点击的 filter chips
3. **区分错误类型**：后端不可达 → "猫窝信号不好"；爬取失败 → "GitHub 大门关了"；LLM 超时 → "猫今天思考太慢"
4. **移动端适配**：猫缩小到 160px，卡片全宽，`max-width: 720px` 在小屏上自然适配

## 七、数据库和接口设计的扩展性

**现状**：所有分析结果塞进 `analysis_json TEXT` 字段，categories 没有独立列。

**后果**：
- 无法用 SQL 直接查询"过去 30 天 AI 类别占比变化"——必须把每行 JSON 都解析出来
- `/api/history` 只返回 `cat_state + narrative`，没有 categories，因为后端不想从 JSON 里再拆一遍

**方向**：
- 将 categories 的 5 个字段拆成独立列（`cat_ai_ml, cat_security, cat_devtools, cat_systems, cat_other`），同时保留 `analysis_json` 存完整 JSON 用于兜底
- `/api/history` 补全 categories 和 dominant_vibe 字段
- 新增 `/api/stats` 聚合接口：近 30 天各分类均值、cat_state 分布饼图数据、最高星数日等

---

## 八、长期产品定位：猫的情绪价值才是核心差异

目前产品回答的问题是："今天技术圈发生了什么？"——但这个问题 GitHub Trending 本身已经回答了。

猫这个创意真正的差异化在于**情绪价值**和**叙事角度**。一个更明确的产品定位：

> **"用一只毒舌猫帮你对抗信息焦虑"** —— 用户不需要读 25 个项目名，看一眼猫就知道今天技术圈在发生什么。

**可考虑的长期方向**：

| 方向 | 描述 |
|---|---|
| **个人化** | 用户选择关注领域（后端/前端/AI），猫的吐槽更精准："你关心的基础设施圈今天也死气沉沉" |
| **分享卡片** | 一键生成"今日技术圈猫评"图片（猫 + 一句吐槽），适合发社交网络 |
| **异常日主动通知** | "今天猫炸毛了！有个项目一天涨了 12k 星" → 邮件/微信推送，这才是用户愿意订阅的理由 |
| **猫的连续剧** | 给猫一个名字和背景故事，每天的 narrative 是连续剧的每日一集，用户追的不是数据，是猫的日常 |
| **多数据源** | 接入 Hacker News、Reddit r/programming、arXiv，猫的视角更全面 |

## 优先级建议

| 优先级 | 事项 | 影响 |
|---|---|---|
| P0 | 关掉 MOCK_LLM，LLM 失败回退昨日真数据 | 不修这个，其他优化都没意义 |
| P0 | 调度器从 4h 改为每天一次 + 重试 | 省钱 + 可靠性 |
| P1 | 历史趋势页面（时间轴 + delta 对比） | 留存的核心功能 |
| P1 | 猫状态加"强度" + 随机微表情 | 解决体验疲劳 |
| P2 | LLM 注入昨日上下文 | 让猫有"灵魂" |
| P2 | 前端加载骨架屏 + 错误分类 + 语言过滤 | 基础体验打磨 |
| P3 | 数据库 categories 拆列 + stats 接口 | 为趋势图打地基 |
| P3 | 规则与叙事分离（代码算比例，LLM 只写吐槽） | 结果稳定性 |
| Future | 个人化、分享卡片、主动通知、多数据源 | 产品差异化 |
